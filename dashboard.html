5 分钟上手（零基础先看这个）

  1. 进项目：终端 cd 你的项目文件夹 → 输入 claude 回车
  2. 说人话：像聊天一样说要做什么，例 帮我把首页标题改成蓝色
  3. 看确认：它要改文件/跑命令时会问你 → y 同意 / n 拒绝 / 直接打字补充
  4. 遇报错：把报错整段贴给它，或说 又报错，帮我查
  5. 换任务：先 /clear 重开；对话太长用 /compact 压缩

💡 口诀：说清楚要什么 + 让它给实跑证据 + 大改动先让它出计划

10 步走通一个真实项目（新人第 1 周照做）

STARTER
  1. 装好 · brew install --cask claude-code 或官网下载 → claude --version 验证
  2. 登录 · claude 进交互 → 浏览器完成 OAuth（OAuth 用官方 Anthropic 账号）
  3. 进项目 · cd ~/projects/myapp → claude → 让它自动建 CLAUDE.md
  4. 写 SPEC · 说「用 AskUserQuestion 面试我，问边界和我没想到的，再写 SPEC.md」
  5. 小步迭代 · 一次只让做一件事，做完看 diff → /clear → 下一件事
  6. 让验证 · 写代码同时让跑测试；说「完成后跑 npm test 验证」
  7. 查错 · 又报错 → 系统化错误分析；或贴报错整段
  8. 升级 · 全面升级 → 自动诊断 + 改配置 + 复核（gated 范式）
  9. 记忆 · 把今天的好经验写进 ~/.claude/CLAUDE.md（永久记忆）
  10. 扩能 · 安装需要的 /plugin 或 /skill → smoke-tests.sh 体检
🎯 核心心法：Claude Code 是「对话式开发伙伴」，不是「一键全做」。你越清楚要什么，它越能帮到你。

高级用法实战示例 — 5 个场景复制即用

5 EXAMPLES

以下示例来自官方文档与实战教程，每例含完整命令、预期输出、适用场景和踩坑提示。点击标题展开即可复用。

▶ 例 1 · /loop 定时监控 — 每 30 分钟巡检服务状态

命令

/loop 每 30 分钟执行一次：
1. 检查 localhost:3000 是否有响应，打印 HTTP 状态码
2. 如果状态码 ≥ 500：输出警告"服务异常"并记录时间戳
3. 如果状态码 200-499：输出"正常"后静默等待
4. 如果网络超时：标记状态，跳过本轮，下轮自动重试

预期输出

[10:00] HTTP 200 — 正常
[10:30] HTTP 200 — 正常
[11:00] HTTP 503 — ⚠ 服务异常 (2026-06-26 11:00:12)

何时用

有明确时间间隔的周期性巡检、监控、数据采集。不需要"完成"，只需要"持续观察"。区分标准：有终止条件 → 用 /goal；只要定时跑 → 用 /loop。

⚠ 踩坑 · /loop 本身只是定时器（每 N 分钟触发一次 prompt），智能决策来自 prompt 里的判断逻辑，不是调度器本身。4 个条件分支必须写进 prompt，否则 Claude 只会报告，不会决策。
▶ 例 2 · /goal 测试修复 — 让测试全绿且 lint 干净

命令

/goal 让 test/auth 下所有测试通过且 lint 干净。
证明方式：npm test test/auth 与 npm run lint 退出码均为 0。
约束：全程不得修改/删除/注释任何测试文件，不能 skip 用例，不得放宽断言。

预期输出

Claude 持续修改生产代码 → 每轮跑 npm test + npm run lint
→ 两者退出码均为 0 时自动停止，汇报修改摘要

何时用

有可测量终态的任务：修好 bug、让测试通过、让 lint 干净、让 CI 变绿。好 goal 公式 = 可测量终态 + 证明方式 + 防失智约束，三要素缺一不可。

⚠ 踩坑 · 缺少"约束"时，Claude 可能通过注释掉断言、把测试 skip 来"达成"目标。必须明确写出禁止什么，才能防止 Claude 走捷径绕过目标。
▶ 例 3 · Dynamic Workflow — 并行审查 80 个 API 端点

命令

ultracode: 审查 src/routes/ 下每个 API 端点
有没有漏掉鉴权检查，输出"端点 → 问题描述"清单，
无问题端点可省略。use 20k tokens

预期输出

Claude 写 JS 编排脚本 → 并行启动数十个 subagent
各自检查一个端点 → 汇总"端点 → 漏洞"报告

何时用

大批量同质任务（审 N 个端点、迁移 200 个文件、扫一批日志）。6 种编排模式：分类路由 / 扇出合成 / 对抗验证 / 生成过滤 / 锦标赛 / 循环直到停。prompt 里点明模式往往比泛泛说"帮我认真做"有效得多。

⚠ 踩坑 · Token 消耗显著高于普通会话；首次触发时 Claude 展示计划并要求确认，先写 use Nk tokens 设预算防失控。小任务（单文件改动）不要开 Workflow，杀鸡用牛刀。
▶ 例 4 · Agent Teams — 三角色并行审查 PR

命令

创建一个 agent team 审查 PR #142，派三个审查者：
- 一个专注安全隐患
- 一个检查性能影响
- 一个核对测试覆盖率
让他们各自审查并抢先汇报发现。

预期输出

三个 teammate 并行审查 → 通过共享 Task List 自协调
→ 各出报告 → Lead 汇总，交叉补位发现单视角漏掉的问题

何时用

多角色需要互相通信协调（写代码 + 审查 + 写文档三者有依赖）；多视角探讨同一问题（UX / 架构 / 唱反调）。前提：开启 CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1（需 v2.1.32+）。

⚠ 踩坑 · 每个 teammate 都是独立 Claude 实例，Token 消耗远超单 session。任务是"互不相关的并行活儿"→ 用 claude agents（Agent View）更省；单聚焦子任务 → 用 Subagent。别为了酷炫开团队。
▶ 例 5 · SkillOpt — 让 AI 自己训练自己的 Skill 文档

核心工作流（Microsoft SkillOpt 范式）

# Step 1 — rollout：用当前 Skill 跑一批真实任务，收集失败样本
# Step 2 — reflection：optimizer(opus) 分析失败模式，提 ≤4 条规则修改
# Step 3 — validation gate：在验证集上评估新版，只有真正提升才保留
# Step 4 — 重复，直到验证集收益为 0

# 对应 OMC Skill 的迭代方式：
# 1. 用 evolution-scout / /evolution 扫本 session 失败案例
# 2. 让 opus 提 ≤4 条规则补丁，写进 .claude/skills/*.md
# 3. 下次实际任务里验证，确认提升才正式保留

论文实测结论

SkillOpt 在 GPT-5.5 直连场景平均提升 +23.5 分；Claude Code 环境 +19.1 分。打败了人类手写 Skill 与所有自动化基线。核心洞察：好 Skill 不靠坐在电脑前"想"出来，而是从失败样本里"训练"出来的。

⚠ 踩坑 · 每轮最多只改 4 条规则。一次大改动会让好坏改动混在一起，验证集无法判断哪条在起作用，导致 Skill 震荡回退。渐进式 + 验证门控是 SkillOpt 稳定的关键。

workflow · 多 agent 确定性编排

L2 编排
场景

同时做几件互不依赖的事——比如同时让 PM 分析需求、架构师出方案、工程师写代码，三人并行，互不等待，最快速完成。

一句话口语触发

「帮我用并行方式同时搞定这几件事」

实际命令
/oh-my-claudecode:ceo-mode 同时做：① 竞品调研 ② 技术可行性方案 ③ MVP 功能清单
# Workflow 由 ceo-mode 自动选用，无需手动触发
预期产出

ceo-mode 判断子任务相互独立 → 自动选 L2 Workflow → 多个专家角色同时并行各干各的 → 全部完成后汇总一份交付报告，比串行快数倍。

适用边界

子任务之间有强依赖（B 必须等 A 结果）→ 改用串行 pipeline；需要组员实时互相讨论修改方向 → 用 team；只有 1 件事 → 直接派单个 executor agent，不用 workflow。

ralph · PRD 持久循环

TIER-4
场景

要完整做完一个功能模块（如用户注册＋邮件验证），有好几个子步骤，中途可能失败，要求"必须全部做完才算数"，不允许半途交差。

一句话口语触发

「ralph 完成这个功能，别停」/ 「keep going 直到全部 pass」/ 「must complete」/ 「don't stop」

实际命令
/oh-my-claudecode:ralph 完成用户注册模块：表单验证 + 密码加密 + 邮件确认，全部通过才停
# 可选：--critic=architect（默认）/ --critic=codex / --no-deslop
预期产出

自动生成 prd.json（拆成若干 user story）→ 逐个实现＋独立 reviewer 验证 → 失败自动重试 → 全部 passes:true 后 architect 最终签字才真正停，进度写入 progress.txt。

适用边界

简单一步修 bug → 直接用 executor（ralph 杀鸡用牛刀）；只想先探索不确定做啥 → /plan；只要并行不要保证完成 → ultrawork；不想让它长跑 → Ctrl+C 随时停。

ultrawork · 并行高吞吐执行

TIER-4
场景

有 3 件互相独立的事要做——比如同时修 3 个不相关的 bug，或同时给 5 个模块补文档，互相无依赖，不想一个个排队等。

「ulw 同时做这几件事」/ 「ultrawork 并行处理」

适用边界

TIER-4
场景

PreToolUse · Bash

它守护什么

每次 Claude 要执行 bash 命令时，这个守卫先扫描命令内容。命中危险模式就立即拦截并告诉你，等你确认或拒绝，不会悄悄跑掉。

拦截的操作类型

rm -rf / rm -r DROP TABLE git push --force git reset --hard kubectl delete shutil.rmtree mv 到 .Trash

拦截时你看到什么

# Claude 想执行这条命令：
rm -rf ./logs/old

# 守卫拦截，你会收到提示：
⚠ 危险操作检测
命令: rm -rf ./logs/old
→ 确认执行 / 跳过 / 修改命令？

你不想要这个守卫时

说 "careful 模式" 可以显式激活更严格版本（gstack-careful skill）。想临时绕过单次拦截，在提示里选"确认执行"即可。

51 个守卫全都是这个逻辑——监听特定事件、扫描特定条件、命中就介入。它们不会影响正常操作速度，只在真正有风险时出声。你唯一需要做的事：看到提示时认真读一下再决定。
