MYCC 能力全景总册（Capability Master Index）

这是什么：你本机 Claude Code（代号 MYCC）当前所有能力的权威清单——装了多少、启用多少、每一类是什么、怎么用。
数据保证：下面每个数字都是 2026-06-17 在本机实跑命令数出来的，不是凭印象（铁律 8）。
唯一权威：本册是能力盘点的 SSoT（Single Source of Truth）。其他手册讲"怎么用"，本册讲"有什么"。

2. 铁律 Iron Laws（12 条 — 行为底座）

每条都有真实 hook 落地，不是口号。完整原文见 rules/mycc-core.md（W4 精简后合并为单文件）。

# 铁律 一句话 落地机制
1 并行优先 ≥2 独立子任务必并行 serial-bash-circuit.sh
2 失败 11 分类 + 自动恢复 失败归类→配方恢复→升级 error-circuit-breaker.sh
3 验证证据 + 防偷懒 + Subagent 闭环 完成必给实跑证据，禁"应该可以了" Stop hook llm-judge-completion.sh
4 记忆持久化 错误/偏好/架构/进度分层落盘 auto-memory
5 官方/已有优先 官方>skill>agent>MCP>重造 四级能力阶梯
6 Subagent 模型显式 model 必传 sonnet/opus/haiku 或注明 inherit 软约束+frontmatter
7 官方直连 禁第三方代理 direct-connect-guard.sh
8 Karpathy 五原则 先读后写/精准改/反推测/声明目标/先验证再断言 uncertainty-detector.sh
9 失败 2 次切方法论 同法连败 2 次强制换思路 error-circuit-breaker.sh
10 Writer ≠ Reviewer 写代码与审代码不同 agent Workflow pipeline 分离
11 澄清优先 + 不可行硬声明 模糊先问（≤2题+示例），不可行给替代 clarity_scorer.py
12 TDD 垂直切片 红→绿→重构，一次一个完整切片 软约束+Gate 2

四步工作流骨架（已合并入 rules/mycc-core.md）：

Step 1 规划 Plan    → ≥3文件或≥5轮触发，产出 plans/{slug}.md
Step 2 并行 Parallel → ≥2独立子任务并行（Workflow/Agent/fork/tmux）
Step 3 强制调试      → 失败≥2次，5思路选1+验证，禁说"做不到"
Step 4 验收闭环      → 每次声称完成，5 道 Gate 门控，实跑证据

3. Hooks 钩子（57 文件 / 48 注册 / 14 类事件 — 自动化规则核心）

Hook 是"由 harness 执行、不是 Claude 自己执行"的自动化——满足"每次 X 就 Y"的规则。

注册的 14 类事件（settings.local.json 实读 · 2026-06-23 · 共 48 挂载点）：

事件 挂载数 典型作用
PreToolUse 9 工具调用前拦截（危险命令阻断、并行提醒、隔离区检查）
SubagentStop 3 子代理结束时（证据校验、模式捕获）
UserPromptSubmit 3 你发消息时（intent 路由、官方源提醒、安全注入）
SessionStart 3 开会话时（CC Switch 护甲、handoff 注入、Semgrep 检查）
PostToolUse 2 工具调用后（结果处理）
Stop 2 主任务停止时（完成质量门控 llm-judge-completion.sh）
PostToolUseFailure 1 工具失败后
PermissionRequest 1 权限请求时
PreCompact / PostCompact 各1 上下文压缩前后（防 resume 污染）
SessionEnd 1 会话结束归档
SubagentStart 1 子代理启动
StopFailure 1 停止失败
MessageDisplay 1 输出展示时（uncertainty-detector.sh 不确定词提醒）

代表性 hook（66 个文件中的关键几个）：
- serial-bash-circuit.sh — 连续串行 Bash 熔断（强制并行，铁律1）
- error-circuit-breaker.sh — 失败计数 + 方法论切换（铁律2/9）
- llm-judge-completion.sh — 完成证据质量校验（铁律3）
- direct-connect-guard.sh — 官方直连守卫（铁律7）
- clarity_scorer.py — 模糊度评分 + 不可行拦截（铁律11）
- block-dangerous-commands.sh — 危险命令阻断（rm -r/rmtree/mv到Trash）
- cleanroom-precheck.sh — 清理类操作前快照（防雪崩）

9. 实战示例 — 微商城项目怎么调动这些能力

完整 816 行实战见 docs/references/XIAOBAI-MALL-MANUAL.md。这里展示一个真实项目如何串起上面所有能力：

阶段 你说 调动的能力 涉及组件
1. 需求 "队长，做个微商城" smart-flow-all + needs-analyst skill + agent
2. 规划 （自动）Step 1 ultraplan / product-manager skill + agent(_core)
3. 设计 （自动）并行 architect + designer agent(_core)
4. 开发 （自动）Step 2 engineer TDD + Workflow 并发 agent + 铁律12
5. 审查 （自动）Step 4 code-reviewer + security-audit agent + skill（铁律10 Writer≠Reviewer）
6. 验收 （自动）Gate 1-5 verifier + Stop hook agent + hook
7. 部署 "上线" team-deploy（蓝绿/回滚） agent(deploy)
8. 归档 （自动）收口 task-closer + Notion MCP agent + MCP

一句话：你只说"队长，做个微商城"，背后是 skill 路由 → agent 编排 → hook 门控 → MCP 落地 → 铁律保障 的全链路自动协作。

10. 版本更新机制（本册如何保持新鲜）

本册采用语义化版本，更新分两档：

大版本（Major，如 V1 → V2）

触发：能力结构性变化——
- 新增/删除一整类组件（如引入新的 MCP 类别、Workflow 体系重构）
- 铁律条数变化（12 → N）
- 插件市场大规模增减（±5 个 marketplace）
- 文档联动关系图变化

动作：重跑全套统计命令（见文末附录）+ 重写受影响章节 + 更新 frontmatter version 主号 + 在 §0 加"数字漂移提醒"。

小版本（Minor，如 V1.0 → V1.1）

触发：数量级微调——
- skill/agent/hook 个数变化（如 125 → 130 skill）
- 启用插件增减（35 → 37）
- 单个组件的用法/触发词更新

动作：只更新 §1 总览速查表对应数字 + frontmatter date + 改动章节，主号不变。

更新触发方式

  • 手动：说 更新能力总册 或 刷新 MYCC 总册
  • 建议节奏：每次"全面升级"（claude-code-better）后顺带刷新；或每月一次
  • 自动候选：可挂到 weekly evolution-scout（进化扫描）后自动比对数字漂移并提示
