## 🤔 Claude Code 是什么？

简单说：**把 Claude AI 直接放进你的终端/编辑器，让它帮你写代码、改 bug、做调研、管理项目**。

不是网页版对话框——它可以直接读你电脑上的文件、执行命令、写代码、运行测试，就像雇了个程序员坐在你旁边随时帮你干活。

加上 **OMC（oh-my-claudecode）** 这套增强系统，还多了 **150 专项技能、50 自动守护钩子**、可以同时派出多个 AI 并行工作。

> **小白比喻**：普通 Claude 网页版像一个顾问，你只能和他聊聊；Claude Code 像把这个顾问接进你公司内网，他能直接翻你的文件柜、在你的电脑上动手改东西、还能帮你雇更多临时工同时干活。

## 🚀 安装和启动

### CLI（命令行版）

```bash
# 安装
npm install -g @anthropic-ai/claude-code

# 启动（在你的项目目录里）
cd ~/projects/my-project
claude
```

第一次启动会让你登录 Anthropic 账号，登录后就能用了。

> **为什么要 cd 进项目目录？** 因为 Claude 会读取你当前目录下的所有文件，在哪个目录启动，它就知道那个项目。就像你告诉新员工"你的工位在这里"，它才知道要看哪些文件。

### Desktop（桌面客户端版）

直接打开 Claude Code 桌面应用，点击 New Project 或者直接拖入项目文件夹。

### 两个版本的区别

| | CLI | Desktop |
|---|---|---|
| 适合 | 长任务、复杂项目、脚本自动化 | 快速问答、轻量改动、可视化 |
| 功能完整度 | 100% | ~70%（部分 hooks/插件不支持）|
| 推荐用法 | 主力工作 | 辅助/临时 |

> **选哪个？** 初学者两个都试试。日常重度使用推荐 CLI，因为钩子（Hooks）和技能（Skills）在 CLI 下功能完整；Desktop 适合快速查个问题或做轻量修改。

## ⚡ 5分钟快速上手

### 第一步：进入项目目录

```bash
cd ~/projects/my-project
claude
```

### 第二步：跟 Claude 说话（就像聊天一样）

```
你：帮我看一下这个项目是做什么的

你：把 src/utils/helper.js 里的 formatDate 函数改成支持时区

你：现在报了一个错：TypeError: Cannot read property 'id' of undefined，帮我找一下哪里有问题

你：帮我给这个项目写一个 README
```

就是这么简单。不需要学特殊语法，说人话就行。

> **为什么说人话就行？** Claude 经过训练，能理解上下文和意图。你不需要像命令行那样精确，"帮我把这个按钮改成蓝色"就够了，不需要写"将 CSS 中 button 选择器的 color 属性改为 #0000FF"。

### 第三步：回应 Claude 的确认请求

Claude 在做某些操作（比如修改文件、执行命令）前会问你：

```
Allow this action?
> Yes (y) / No (n) / Always allow (a)
```

- 按 `Y` 或回车 = 允许这一次
- 按 `A` = 以后同类操作都允许
- 按 `N` = 不允许

嫌烦的话，可以说"**火力全开**"，之后就不再问了，全自动干活。

> **什么时候用 A（永久允许）？** 比如 Claude 要运行 `ls` 查看文件列表，这类只读操作你可以永久允许。而"删除文件""执行部署"这类高风险操作，建议每次都手动确认。

## 🧠 几个核心概念（用大白话理解）

### 会话（Session）
每次 `claude` 启动就是一个新会话，Claude 只记得这次对话里的内容。说 `/clear` 就是清空重开，换个话题时建议用。

> **比喻**：就像每次打开一个新的聊天窗口，Claude 不会自动记得上次说过的事。但是有了 OMC 的**多层记忆系统**（见后文），你说"记住这件事"它会跨会话记住。

### 技能（Skills）
可以理解为"特定领域的专家模式"。比如 `/code-review` 是代码审查专家，`/design` 是 UI 设计专家。用 `/技能名` 或者口语关键词触发。

> **比喻**：普通 Claude 是全能助理，技能相当于给它戴上"专业帽子"。戴上代码审查帽，它就会特别严格地挑代码问题；戴上 PM 帽，它就会从产品视角给建议。

### 钩子（Hooks）
在后台默默运行的守护程序，你不用管它。比如你一发消息，系统自动帮你检索相关历史记忆；你改了 settings.json，自动检测是不是有安全风险。

> **比喻**：钩子像保安 + 助理的组合。你每次进门（发消息），保安自动核查（检测危险操作），助理自动准备好相关资料（检索记忆）。这些都在你不知情的情况下完成。

### Agents
当你说"队长，帮我做 XX"，Claude 会自动拆成多个任务，同时派出多个 AI 并行工作，最后汇总结果。速度是单线程的好几倍。

> **比喻**：一个人干活 vs 派出一个项目组。Claude 是总指挥，它拆分任务后，同时让"前端 AI"、"后端 AI"、"测试 AI"各干各的，最后汇总。你只需要等最终结果，中间过程它帮你协调。

## 💬 最常用的口语说法（直接抄就能用）

### 开始一个任务

```
"帮我做一个商品详情页"
"给这个函数加单元测试"
"把登录功能重构一下，现在代码太乱了"
"帮我看看这个项目结构，然后给一个优化建议"
```

### 触发特殊模式

```
"队长，把整个购物车模块做完"        → 自动四步工作流，更细致、有验证
"火力全开，把剩下的 TODO 全做完"    → 极限并发，全自动不打断
"帮我调研一下 tRPC 和 REST 的区别"  → 自动深度调研多渠道
"做完之后做个全面的代码审查"        → 审查+修复
```

### 调试报错

```
"这个地方报了 404，帮我查一下"
"又报错了：[把报错信息粘贴进来]"
"反复出这个错，系统排查一下根因"
```

> **小白技巧**：报错信息直接完整粘贴，不要自己总结。Claude 需要看完整报错堆栈才能准确定位问题，你总结可能会丢掉关键信息。

### 让 Claude 记住事情

```
"记住，这个项目用的是 PostgreSQL 14，不是 MySQL"
"记一下，API 超时时间统一设 30 秒"
```

> **这条会真正保存**：说"记住"后，系统会把这条信息写入持久记忆文件，下次新开会话仍然有效。不只是这次对话的便条，而是长期有效的笔记本。

### 撤回和修改

```
"这个方向不对，撤回去重新做"        → 说这句话或按 Esc Esc
"重新考虑一下，用另一种方式实现"
```

## 🛡️ 权限确认怎么回

Claude 修改文件或运行命令前，会有这样的提示：

```
◆ Allow bash command?
  ls -la src/
  Yes / No (always skip with "No")
```

**常见场景：**

| 情况 | 怎么回 |
|---|---|
| 正常操作，允许 | `Y` 或回车 |
| 这个不要执行 | `N` |
| 全部自动允许 | 说"**火力全开**"或提前说"**YOLO**" |
| 不确定，先看看 | `N`，然后问 Claude "你要执行什么，先说一下" |

> **什么是 YOLO？** "You Only Live Once"——在 Claude Code 里表示"放开限制，全自动跑，别问我"。适合你已经信任 Claude、想让它连续完成一大堆操作的时候。危险操作（删文件、部署到生产）不建议用 YOLO。

## 📟 常用内置命令详解

### `/clear` — 清空重开
用法：直接说 `/clear`
场景：换了个完全不相关的任务，或者对话太长太乱了
效果：清空所有对话历史，上下文完全重置

> **什么时候用**：比如你刚做完登录功能，现在要做支付功能，两个完全无关。这时清空，Claude 就不会把登录的记忆混入支付任务里。

---

### `/compact` — 压缩上下文
用法：`/compact`
场景：对话已经很长了，Claude 开始忘前面说的事
效果：把对话压缩成摘要，腾出空间继续

> **什么时候知道要用**：如果你发现 Claude 开始重复已经做过的事，或者问你已经告诉过它的信息，就说明上下文窗口快满了，该用 `/compact` 了。或者直接在长 session 开始时先主动 `/compact`，防患于未然。

---

### `/model` — 切换模型
用法：`/model opus`  `/model sonnet`  `/model haiku`
- opus：最强但慢，适合复杂架构/深度任务
- sonnet：平衡，日常主力
- haiku：最快，适合简单查询

> **小白选模型指南**：不知道用哪个，就用默认的 sonnet。如果任务很重要/很复杂（比如设计整个系统架构），换 opus。如果只是问个小问题（"这个函数叫什么"），haiku 更快。

---

### `/goal` — 设定任务完成标准
用法：`/goal "购物车可以正常加减商品并显示总价"`
场景：长任务（做超过5轮的）开始前说，防止跑偏
效果：Claude 会一直拿这个标准来检查自己做完没有

> **为什么重要**：没有目标的话，Claude 可能做到一半觉得"差不多了"就停了，但实际还没满足你的要求。设定 goal 就是给它一个明确的终点线。

---

### `/btw` — 侧问（不影响主任务）
用法：`/btw 这个组件的名字叫什么？`
场景：正在做一个大任务，突然想问个小问题
效果：Claude 回答这个问题，但**不会把它算进主对话**，不污染上下文

> **比喻**：像你在会议室开会（主任务），突然想问旁边同事一个小问题（/btw），不用打断整个会议，问完继续。

---

### `/cd` — 切换工作目录
用法：`/cd ~/projects/backend`
场景：同一个 session 里需要切到另一个项目
效果：切换目录但保留对话历史和缓存（比退出重开省多了）

---

### `/safe-mode` — 纯净模式
用法：启动时加参数：`claude --safe-mode`
场景：Claude 出现奇怪行为、hook 报错、插件冲突时排障
效果：禁用所有 CLAUDE.md、插件、skills、hooks、MCP，用最干净的 Claude

> **什么时候用**：系统行为变奇怪了，怀疑是某个自定义配置搞的鬼，用 safe-mode 排查——如果 safe-mode 下正常，说明是配置问题，不是 Claude 本身的问题。

---

### `/rewind` — 撤回
用法：`/rewind`
场景：Claude 改了一堆代码，发现方向错了
效果：回退到上一个检查点（Claude 做大操作前会自动打 checkpoint）

> **小白注意**：rewind 是回退代码修改，不是回退对话。如果 Claude 改了20个文件但方向全错了，`/rewind` 可以一键全部撤销，不用一个个手动改回去。

---

### `/code-review` — 代码审查
用法：
- `/code-review` — 只看问题，不修改
- `/code-review --fix` — 看完直接修复
- `/code-review --comment` — 看完发 PR 评论
- `/code-review ultra` — 云端多 agent 深度审查

> **日常建议**：每次做完一个功能，都跑一次 `/code-review --fix`，让 Claude 帮你找并修掉明显问题。这比自己盯着看要快得多，而且不会漏掉常见的安全漏洞。

## 🔥 四大常用工作流程

### 流程1：修 Bug

```
1. 把报错信息直接粘给 Claude
   "这里报了：[报错内容]，帮我找一下"

2. Claude 会读相关文件，定位根因

3. 问 Claude：
   "你打算怎么改？先说方案"

4. 确认方案没问题后：
   "好，按这个改"

5. 改完问：
   "改完了，帮我验证一下这个场景还能正常跑"
```

> **为什么第3步要先问方案？** 防止 Claude 改了一大堆，结果方向你不认可，白做了。让它先说再做，你确认方向对了再动手，是处理复杂 bug 的最佳实践。

### 流程2：开发新功能

```
1. 描述需求：
   "帮我做一个商品详情页，要有图片轮播、规格选择、加入购物车"

2. Claude 会先问一些关键问题（什么技术栈、什么框架等）

3. 确认技术细节后它开始写代码

4. 做完后说：
   "帮我做个代码审查"  （/code-review --fix）

5. 有问题自动修掉，没问题收工
```

> **需求说得越具体越好**：比如"商品详情页要跟我们现有的 ProductCard 组件风格一致，用 Tailwind 写，不要 antd"。这样 Claude 就不用猜，直接按你的规范来。

### 流程3：做调研

```
1. 说：
   "帮我调研一下支付宝和微信支付接入，选哪个更合适，
    我们是 Node.js 后端，交易量初期不大"

2. Claude 会用 deep-research 多渠道调研

3. 给你一份有来源、有对比、有★推荐的报告

4. 你说："好，就按推荐方案来，给我接入教程"
```

### 流程4：全面审计/升级

```
1. 说：
   "帮我做一次全面的配置审计"  → 触发 /audit-all-better

2. 或者：
   "全面升级一下"              → 触发 /claude-code-better

3. 看报告，Claude 会给出 P0/P1/P2 分级的问题清单

4. P0 问题（严重）自动修，P1/P2 问你要不要改
```

> **P0/P1/P2 是什么意思？** P0 = 最严重，必须立即修（比如安全漏洞、系统崩溃风险）。P1 = 重要但不紧急（性能问题、代码质量差）。P2 = 建议改进（规范性问题、小优化）。Claude 自动帮你分级，你只需要决定 P1/P2 改不改。

## 🪢 高频入口按场景串起来（2026-06-25 增 · 来自 dashboard.html §C7）

> 📌 **场景化思维** · 别死记命令，按你"想干什么"挑入口。每个入口都有完整 4 步链路，对应不同工作流。

### 🎯 我想"先摸清情况再动手" → `全面体检 / 系统体检`
**对应入口**：`/full-checkup`（gated 范式）
**完整链路**：
1. **只读诊断（自动）** · Claude 自己扫你的配置/能力/健康度，不动任何东西
2. **HITL 确认门（必问）** · Claude 把发现的问题列出来，问你改哪些、改什么
3. **mutation（人批后才动）** · 改前先 snapshot 可回滚，触 hook/cron 单独再确认
4. **复核对比** · 改完跑一遍同样诊断，验证修复真生效
> 💡 **适合场景** · 不确定哪里要改 · 升级前先体检 · 月度/季度巡检

### 🔥 我想"升级 CC 本体" → `全面升级 / 进化一下 / 自我进化`
**对应入口**：`/claude-code-better`
**完整链路**：
1. **检查新版变更** · 拉官方 changelog + 社区 ytb/reddit
2. **对比我机现状** · 我这版是什么、缺什么
3. **异构对抗判定** · 5 联 AI（kimi/mmx/opencode/qwen/mavis）各投一票要不要改
4. **改造 + 回归验证** · 真的改完后跑现有测试，确保没坏
> 💡 **适合场景** · 看到 CC 升级弹窗 · 新功能觉得有用 · 想保持工程化

### 🪜 我想"复杂任务总编排" → `队长 / 火力全开 / 活力全开 / YOLO`

**完整链路**：
1. **需求梳理** · 把"我要做 X"拆成子任务
2. **派 subagent 并行** · 多个子任务同时跑（节省时间）
3. **顾问对抗裁决** · 多顾问各给方案，主调度器裁决
4. **Writer≠Reviewer 收口** · 写的人不能自己审，换 agent 审
> 💡 **适合场景** · 大改动 · 跨多个文件 · 不确定从哪开始 · 想让 Claude 极限输出

3. **对抗裁决** · 顾问间互相挑刺
4. **综合报告** · 出一份可执行结论
> 💡 **适合场景** · 投资前 · 立项前 · 跨境业务前

### 🔍 我想"调研/对比/选型" → `调研 / 深度调研 / 对比 / 选型 / 横评`

3. **blast radius 评估** · 影响多大、有没有连锁

## 🆘 遇到问题怎么办

### Claude 卡住不动了
按 `Esc Esc`（连按两次），中断当前操作。然后说说你想换个方向。

### Claude 方向跑偏了
```
"停一下，这个方向不对。我要的是 X，不是 Y。重新来"
```

> **小白常见坑**：很多人觉得打断 Claude 不好意思，让它一直做下去，结果做了100行全是错的。**你随时可以打断**，这是正常操作，不是在为难 Claude。

### Claude 改了一堆错的东西
```
/rewind
```
或者说：`"撤回所有修改，回到你开始之前的状态"`

### 对话太长 Claude 开始犯傻
```
/compact
```
或者干脆 `/clear` 开新会话，把关键背景重新说一遍。

### Hook 报错、行为异常
```
claude --safe-mode
```
在纯净模式下复现问题，判断是配置问题还是代码问题。

### 想问一个问题但不想影响当前任务
```
/btw [你的问题]
```

## 🎓 进阶：理解"四步工作流"

当你说"**队长**"，Claude 进入完整工作流：

1. **Plan（规划）** — 先理解需求，列出要改哪些文件、怎么改、验证方法
2. **Parallel（并行执行）** — 能同时做的任务自动并行，速度快几倍
3. **Force-Solve（强制解决）** — 遇到问题不放弃，最少尝试5种思路
4. **Verify（验证收口）** — 做完给实际运行证据，不说"应该可以了"

不说"队长"就是普通对话模式，适合简单任务。

> **小白理解**：说"队长"相当于告诉 Claude"这件事很重要，我要你认真对待，不要偷懒"。它会更谨慎地规划、验证，最后给你实际跑过的证据，而不是猜测。普通任务就不用，免得大材小用。

## 🆕 最新工程化能力（大白话版）

> 这一节把 OMC 系统的高级能力翻译成小白能懂、能直接用的语言。每个能力都附"一句话是什么 + 怎么触发 + 什么时候用"。

---

### 1. 异构对抗：让其他 AI 帮你挑错

**一句话是什么：**
用 kimi、mmx、qwen、opencode 等非 Claude 模型，从反方角度挑你方案的毛病。就像你设计了一个方案，再去找几个"挑剔的外人"质疑它——他们会发现你自己看不到的盲点。

**怎么触发：**
```bash
ai-panel "你的问题或方案描述"
```

**小白什么时候用：**

在重大决定前用一句话问一下。比如：
```bash
ai-panel "我们要用 MongoDB 还是 PostgreSQL 做主数据库，初期用户量1万，数据结构比较灵活"
```
它会让多个 AI 同时从不同角度分析，给你一个更全面的视角，避免只听 Claude 一家之言。

> **为什么有用**：每个 AI 模型都有自己的训练偏好和盲区。Claude 可能对某个方案特别看好，但 kimi 或 qwen 会从不同角度挑战它。多模型碰撞 = 更健壮的决策。

---

### 2. 队长模式：极限全权统筹

**一句话是什么：**
说"队长"或"火力全开"，Claude 进入最高配置的工作模式：先规划、后并行执行、遇到问题死磕不放弃、最后给你实跑证据。

**怎么触发：**
```
"队长，帮我把登录模块做完"
"火力全开，把所有 TODO 清掉"
"老大，这个项目全权交给你了"
```

**小白什么时候用：**

任务比较大、比较重要、你希望 Claude 认真对待的时候。比如：
- "队长，帮我做整个用户权限系统，包括登录、注册、JWT、权限控制"
- "火力全开，把这个项目的安全漏洞全修了"

> **普通模式 vs 队长模式的区别**：普通模式 Claude 可能写完就说"好了"，但实际没跑通。队长模式它会强制验证，给你实际运行的截图或输出，不会假完成。

---

### 3. 要求实跑证据（Verify-Before-Assert）

**一句话是什么：**
Claude 有个内置铁律：下结论前必须先实测，不能凭感觉猜。你可以主动触发这个模式，让它给你"实跑证据"而不是"我猜应该可以"。

**怎么触发：**
```
"给我实跑证据，别猜"
"你跑一下看看结果是什么"
"不要告诉我'应该可以'，帮我验证一下"
```

**小白什么时候用：**

当 Claude 说"这个应该能跑""理论上没问题"这类话时，追一句：
```
"别猜，帮我实际跑一下，把结果给我看"
```

> **为什么重要**：AI 会"hallucinate"（幻觉），说得很自信但其实是错的。要求它实跑，就是逼它用真实结果说话，而不是凭印象。这是使用 AI 编程的核心防坑技巧之一。

### 6. 自动回滚：配置坏了一键恢复

**一句话是什么：**
如果 Claude 帮你改配置改坏了，或者你自己操作失误，这个命令可以从自动快照恢复到正常状态。默认是"dry-run"（预演模式），先让你看看会恢复什么，确认了再真正执行。

**怎么触发：**
```bash
# 预览会恢复什么（安全，不会真的改动）
bash ~/.claude/lib/auto-rollback.sh

# 确认后真正执行回滚

```

**小白什么时候用：**

- 升级或修改配置后系统行为变奇怪了
- Hook 报错找不到原因
- 不小心改了配置文件想撤回
- 系统管理员（Secretary）提示"配置漂移"时

> **为什么默认 dry-run 很重要**：先预览、再执行，是安全操作的铁律。你可以先看"它要恢复哪些文件、恢复成什么样"，确认没问题再按确认。不会因为一个命令把好的配置搞坏。

### 7. 风险分级（L0-L3）：小改直做，大改先确认

**一句话是什么：**
系统会自动判断每个操作的风险等级，小操作直接做，大操作（尤其是删除、改配置、部署）会先给你计划，等你点头再动手。

**怎么触发：**

这个是自动的，不需要手动触发。但你可以理解分级含义：

| 等级 | 触发条件 | 系统做什么 |
|---|---|---|
| L0 | 1-2个文件，可撤销，不碰配置 | 直接做，完成后告诉你 |
| L1 | 5个文件以内，可撤销 | 做完自动备份，提交记录 |
| L2 | 超过5个文件，或涉及推送/外部服务 | 先给你一句话简报再开始 |
| L3 | 不可逆操作、改配置/钩子、安全相关 | 先给完整计划，等你确认再动 |

**小白什么时候用：**

你不需要记这些等级，只需要知道：**如果 Claude 跟你确认计划，那是因为这个操作比较重要，认真看一眼再点头**。不要因为嫌麻烦就随便说"是"，尤其是涉及删除、部署、改配置的操作。

> **实际例子**：你说"帮我重构登录模块"（5个文件以上，L2），Claude 会先说"我打算改这些文件、顺序是这样、验证方式是这个"，你说"好"之后它才动手。这能防止 Claude 自作主张做了你不想要的事。

### 8. 定时秘书：自动巡检 + 每日简报

**一句话是什么：**
系统内置了"定时秘书"——每天/每周自动跑维护任务，开 session 时自动提示今天有什么需要关注的。你不需要手动管，它自动在后台帮你盯着。

**怎么触发：**

自动触发，每次开 session 会有提示。如果没有提示，说明没有需要关注的事项。

手动查看：
```bash
# 查看今天的每日简报
cat ~/.claude/reports/daily-brief-$(date +%Y-%m-%d).md

# 查看系统整体健康
python3 ~/.claude/lib/secretary-orchestrator.py report
```

**小白什么时候用：**

- 每次开始工作前看一眼 session 开始时的提示
- 如果提示说"有 X 个钩子冲突"或"baseline 漂移"，在开始正式任务前先处理这些

> **比喻**：就像早上起来手机给你推送"今天有3个日历提醒、1条未读重要消息"，定时秘书做的是同样的事——把需要你注意的系统状态提前告诉你，不让小问题积累成大问题。
