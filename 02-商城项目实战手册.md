# 用 Claude Code 从零构建商城项目——实战手册
> 技术栈：Next.js 14 + Node.js + PostgreSQL + Tailwind CSS
> 适合：有开发基础、第一次用 Claude Code 做完整项目的工程师

---

## 📋 项目概览

**潮牌商城（TrendMall）**
- 前台：商品浏览、搜索、商品详情、购物车、结账、订单查询
- 后台：商品管理、订单管理、用户管理
- 技术栈：Next.js 14 (App Router) + Tailwind CSS + Prisma + PostgreSQL

**用 Claude Code 能做什么：**
- 从需求分析到代码部署，全程 AI 辅助
- 根据你的描述自动生成代码、写测试、做代码审查
- 遇到 bug 自动排查根因、自动修复
- 调研技术方案、比对选型、给出推荐

## 💻 第3阶段：功能开发（按功能循环）

每个功能的标准开发循环：
**需求描述 → 方案确认 → 编写代码 → 代码审查 → 验证**

---

### 示例A：商品列表页（基础功能）

**你说：**
```
"帮我开发商品列表页。
 功能：展示商品卡片（图片、名称、价格、销量），
 支持按分类筛选、价格排序，底部分页。
 数据从 /api/products 接口获取，
 用 TailwindCSS 写样式，要有 loading skeleton"
```

**Claude 会：**
1. 分析需求，列出要创建/修改的文件
2. 问确认：「接口格式是什么？分页是前端还是后端？」
3. 你确认后开始写
4. 写完说：`/code-review --fix` 自动审查+修复

---

### 示例B：购物车功能（状态管理）

**你说：**
```
"开发购物车功能。
 要求：
 - 支持加入购物车、修改数量、删除商品
 - 购物车数据用 Zustand 管理（本地 + 登录后同步服务端）
 - 右上角 badge 显示购物车数量
 - 购物车抽屉侧边弹出
 做之前先给我看一下你打算怎么设计状态管理的方案"
```

> 💡 加"做之前先给我看方案"是好习惯，尤其是复杂功能，确认方向再动手。

---

### 示例C：支付功能（高风险，需要谨慎）

**你说：**
```
"接入微信支付。先帮我调研一下微信支付 JSAPI 和 Native

```


做完后：
```

```

---

### 示例D：后台管理（复杂，用队长模式）

**你说：**
```
"队长，帮我开发后台商品管理模块。
 功能包括：商品列表（搜索、筛选、批量操作）、
 商品新增/编辑（多规格SKU、图片上传到OSS）、
 上下架操作、库存管理。
 用 React 的 ant design 5 做 UI"
```

触发"队长"模式后，Claude 会：
1. 先分析拆分成多个子任务
2. 能并行的任务同时做（列表页、表单页、API 接口可以并行）
3. 每个子任务完成后做验证
4. 全部完成后给汇总报告

---

### TDD 开发模式（写测试再写代码）

**适合：** 核心业务逻辑、支付计算、库存扣减等不能出错的功能

**你说：**
```
"用 TDD 模式开发订单价格计算逻辑。
 规则：商品单价 × 数量 + 运费，
 满 199 免运费，优惠券直减，最终价格不能低于 1 分钱"
```

Claude 会严格走：
1. **红**：先写失败测试（覆盖各种边界情况）
2. **绿**：写最小实现让测试通过
3. **重构**：整理代码，测试仍然通过

### CLAUDE.md 写什么（项目根目录）

```markdown
# TrendMall 商城

## 技术栈
- 前端：Next.js 14 App Router + Tailwind CSS + Zustand
- 后端：Next.js API Routes + Prisma
- 数据库：PostgreSQL 15
- 部署：Vercel + Railway

## 常用命令
- 启动开发：`npm run dev`
- 数据库迁移：`npx prisma migrate dev`
- 生成类型：`npx prisma generate`
- 跑测试：`npm test`

## 关键约定
- API 路径统一加 /api 前缀
- 后台路由加 /admin 前缀
- 错误统一用 { code, message, data } 格式返回
- 所有金额单位：分（避免浮点精度问题）
- 图片存 OSS，数据库只存 URL

## 环境变量
- DATABASE_URL：PostgreSQL 连接串
- REDIS_URL：Redis 连接串
- WECHATPAY_MCHID：微信商户号
- OSS_BUCKET：图片桶名称
```

## 🚨 避坑指南

### 坑1：直接说"帮我做个商城"
❌ 太模糊，Claude 会做出一个跟你想的完全不同的东西

✅ 正确做法：先需求分析（用 `deep-interview`），然后一个功能一个功能做

---

### 坑2：让 Claude 一次做太多
❌ "帮我把整个商城前端都做完"

✅ 正确做法：按功能模块拆分，每次做一个，验证完再下一个

---

### 坑3：代码改乱了不知道怎么撤回
遇到这种情况：
```
/rewind
```
或者：`"把你刚才的改动都撤回"`

也可以用 git 撤回（如果有 commit 的话）：
```bash
git checkout .
```

---

### 坑4：Claude 把测试删掉/改掉让测试通过
如果你发现 Claude 改了测试而不是改实现：
```
"不对，测试不能改。你要修的是实现代码，不是测试。撤回重来"
```

Claude 有内置规则禁止这样做，但如果出现了，这样纠正。

---

### 坑5：Claude 说"应该可以了"但没验证
```
"帮我实际运行一下，给我看运行结果"
```

要求 Claude 给实际执行的证据，不接受"应该没问题"。

---

### 坑6：同一个 bug 改了好几次还没修好
```
"同一个问题改了好几次还没修好，
 帮我系统性排查一下根因，
 不要先修，先把根因分析清楚再说方案"
```

或者触发：
```
"又报错了"  → 触发 systematic-error-analysis
```

---

### 坑7：上下文塞满了无关内容
长期开发同一个项目，推荐每天/每个大模块 `/clear` 开新会话，
把当前关键背景（技术栈、当前在做什么）简单说一遍，比在一个会话里撑几百轮效果好。

```
Week 1：基础框架
  Day1-2：初始化（/init），数据库设计，Prisma Schema

  Day3：结账流程（地址管理 + 运费计算）
  Day4-5：微信支付接入 + 安全审计

Week 4：订单/后台

```

## 🗣️ 完整口语对话示例集

### 示例1：开发一个新功能
```
你："帮我做商品详情页。

Claude：好，我需要确认几点：
    1. 规格组合（颜色+尺码）是否有独立 SKU 和库存？
    2. 图片来自 OSS，URL 格式是？
    3. 加入购物车是直接加还是要登录验证？

你："是的有独立SKU，图片 URL 是 https://oss.example.com/图片名，
    加入购物车需要登录"

Claude：[开始写代码，给出文件列表和实现思路]
    ...写完后...
    已完成，我来做个代码审查

你："好，审查完顺便把代码简化一下"

Claude：/code-review --fix [自动审查修复]
        /simplify [自动简化]
```

### 示例2：排查 bug
```
你："订单提交一直报 500 错误，帮我查一下。
    报错：Internal Server Error，后台日志：
    PrismaClientKnownRequestError: Unique constraint failed on the fields: (`order_no`)"

Claude：找到了，order_no 生成逻辑有问题，高并发时可能重复。
    当前用的是 Date.now() + random，
    应该用雪花算法或者数据库 sequence。
    我用数据库 sequence 来修，更简单可靠。
    你确认要用这个方案吗？

你："好，用 sequence 修"

Claude：[修复，然后给验证方法]
    修完了，可以这样测试：同时提交10个订单，
    看 order_no 是否全部唯一
```

### 示例3：技术调研
```
你："帮我调研一下商城的搜索功能怎么做。
```
