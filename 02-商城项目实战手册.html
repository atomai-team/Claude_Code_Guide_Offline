用 Claude Code 从零构建商城项目——实战手册

技术栈：Next.js 14 + Node.js + PostgreSQL + Tailwind CSS
适合：零基础到有开发基础、第一次用 Claude Code 做完整项目的工程师

怎么用这本手册：从上到下按顺序读。每一步都有：你说什么 → Claude 做什么 → 看到什么算成功 → 卡住了怎么办。照着复制粘贴就能跑起来。

📋 项目概览

潮牌商城（TrendMall）
- 前台：商品浏览、搜索、商品详情、购物车、结账、订单查询
- 后台：商品管理、订单管理、用户管理
- 技术栈：Next.js 14 (App Router) + Tailwind CSS + Prisma + PostgreSQL

用 Claude Code 能做什么：
- 从需求分析到代码部署，全程 AI 辅助
- 根据你的描述自动生成代码、写测试、做代码审查
- 遇到 bug 自动排查根因、自动修复
- 调研技术方案、比对选型、给出推荐

小白须知：
- Claude Code 是一个跑在终端（命令行）里的 AI 工具，不是网页。
- 你需要先安装 Node.js（https://nodejs.org）和 Claude Code（npm install -g @anthropic-ai/claude-code）。
- 安装好后，在终端里输入 claude 就进入对话模式，然后用中文直接跟它说话就行。
- 你说的每一句话，Claude 都会读懂并执行，不需要写代码命令。

🩺 先体检再开工（2026-06-25 增 · 项目实战前必读）

💡 开工前先体检 · 商城项目动辄几十个文件 + 数据库 + 部署，开工前花 5 分钟"体检"能省后面 5 小时踩坑。

🏁 第0阶段：项目初始化（必做）

这一阶段目标：创建项目文件夹，让 Claude 认识你的项目，以后每次开对话不用重复解释。

第一步：创建项目文件夹并进入

在终端输入：

mkdir -p ~/projects/trendmall
cd ~/projects/trendmall
claude

看到什么算成功：终端出现 Claude 的对话提示符（通常是 >），等你输入。

卡住了：如果报错 command not found: claude，说明没安装好，运行 npm install -g @anthropic-ai/claude-code 重装。

第二步：初始化 CLAUDE.md 项目说明文件

进入对话后说：

/init

Claude 会做什么：扫描项目结构，自动生成 CLAUDE.md 文件，记录：
- 项目架构
- 技术栈
- 常用命令（npm run dev、npx prisma migrate 等）
- 特殊注意事项

看到什么算成功：Claude 说"已生成 CLAUDE.md"，你可以用 cat CLAUDE.md 看内容。

卡住了：如果目录是空的，Claude 会生成一个基础版，后面你再补充内容就行。

💡 CLAUDE.md 是 Claude 每次启动都会读的"项目说明书"，写得越详细，Claude 越少问你重复问题。

第三步：深度初始化项目理解

如果是接手别人的项目，说：

/deepinit

或者：

帮我深度理解这个项目的结构、技术栈、业务逻辑，
然后给我一份完整的项目概览

Claude 会做什么：读取所有文件，分析代码结构，用中文给你写一份"项目地图"，说清楚每个目录是干什么的。

看到什么算成功：Claude 输出一个包含目录结构、主要模块、技术栈的总结。

第四步：设置项目记忆

告诉 Claude 项目关键信息（它会记住，下次打开不用再说）：

记住以下项目信息：
- 数据库：PostgreSQL 15，连接串在 .env 的 DATABASE_URL
- 部署：Vercel（前端）+ Railway（数据库）
- 设计规范：主色 #1A1A2E，字体 Inter
- 支付：接入微信支付，SDK 在 /lib/wechatpay.ts
- 后台路由统一加 /admin 前缀

Claude 会做什么：把这些信息存入项目记忆，以后新开对话也记得。

看到什么算成功：Claude 说"已记住以下信息：……"并复述你说的内容。

卡住了：如果 Claude 说没有记忆功能，直接把这段话加到 CLAUDE.md 文件里，效果一样。

💡 第1阶段：需求分析

这一阶段目标：把脑子里模糊的想法，变成一份清晰的功能清单和优先级。

场景：你有一个模糊的想法，需要梳理清楚

你说：

我想做一个潮牌电商，主要卖国内潮牌的衣服鞋子，
目标用户是 18-30 岁年轻人，先做 MVP。
帮我梳理一下需要做哪些功能，排个优先级

Claude 会做什么：
- 引导你回答关键问题（用户规模、支付方式、是否需要库存管理等）
- 生成功能清单 + RICE 优先级排序
- 划定 MVP 范围

看到什么算成功：Claude 输出一个表格或列表，有功能名称、优先级（高/中/低）、原因说明。

卡住了：如果 Claude 问了很多问题让你不知道怎么回答，就说"先按最简单的 MVP 来，其他的后面再说"。

用 deep-interview 深度挖需求

/deep-interview

Claude 会用结构化的方式访谈你，挖出真实需求。适合产品方向还不明确时。

看到什么算成功：Claude 开始问你一连串问题，比如"你的目标用户每周购物几次？""现有竞品是谁？"等，你如实回答就行。

小白提示：这个命令特别适合你脑子里只有一个大概想法的时候，让 Claude 帮你想清楚。

生成 PRD 文档

需求梳理完后：

根据我们讨论的内容，帮我生成一份正式的 PRD 文档，
包含功能列表、用户故事、验收标准

或者用：

/to-prd

Claude 会做什么：把你们的讨论整理成标准格式的产品需求文档，每个功能点有"谁、在什么情况下、做什么、结果是什么"。

看到什么算成功：Claude 输出一份带标题、功能列表、用户故事（"作为……我想……以便……"格式）的文档。

商业可行性分析（可选）

/ceo-mode

Claude 会从市场分析、竞品对标、财务预测等维度帮你判断这个方向是否值得做。

🔧 第2阶段：技术选型和架构设计

这一阶段目标：确定用什么技术来做，设计数据库，画出系统结构图。

调研技术栈

你说：

帮我调研一下做电商前端，Next.js 14 和 Remix 哪个更合适。
我的场景是：商品列表需要 SEO，用户量初期不大，
团队熟悉 React，不想花太多时间在框架上

Claude 做什么：
触发 deep-research，调研官方文档、社区讨论、实际案例，
给出对比表 + ★推荐 + 理由。

看到什么算成功：Claude 给出一个对比表（两个框架各自的优缺点），最后有一个明确的 ★推荐，不会模糊地说"都可以"。

卡住了：如果 Claude 给的结论你看不懂，就说"用大白话再解释一遍，不要用技术术语"。

技术架构设计

根据我们确定的需求，帮我设计整体技术架构。
要包含：前端架构、后端服务分层、数据库设计、
第三方服务接入（支付/OSS/短信），以及部署方案

Claude 会输出：
- 架构图（文字版/Mermaid）
- 关键技术决策说明
- 各层接口设计

看到什么算成功：Claude 给出一个分层的架构描述，能看出"用户 → 前端 → 后端 → 数据库"的数据流向。

生成架构图

/baoyu-diagram

或者：

把架构用 Mermaid 图表示出来

小白提示：Mermaid 图是一种用文字描述的流程图，你把 Claude 给的代码粘到 https://mermaid.live 就能看到图形效果。

数据库设计

帮我设计商城的数据库 Schema。
包含：用户、商品（多规格/SKU）、购物车、订单、支付记录。
用 Prisma Schema 格式输出，注意加合理的索引

Claude 会做什么：给你一份完整的 schema.prisma 文件内容，包含所有表的字段、类型、关系和索引。

看到什么算成功：Claude 输出包含 model User、model Product、model Order 等的代码块。

卡住了：如果你不懂 Prisma 是什么，就说"先解释一下 Prisma 是什么，再给我 Schema"。Prisma 是一个让你用 JavaScript 操作数据库的工具，不用写 SQL。

💻 第3阶段：功能开发（按功能循环）

这一阶段目标：一个功能一个功能地做，每次做完验证，再做下一个。

每个功能的标准开发循环：
需求描述 → 方案确认 → 编写代码 → 代码审查 → 验证

小白须知：不要一次让 Claude 做太多功能，每次只说一个，做完验证没问题再做下一个。

示例A：商品列表页（基础功能）

你说：

帮我开发商品列表页。
功能：展示商品卡片（图片、名称、价格、销量），
支持按分类筛选、价格排序，底部分页。
数据从 /api/products 接口获取，
用 TailwindCSS 写样式，要有 loading skeleton

Claude 会：
1. 分析需求，列出要创建/修改的文件
2. 问确认：「接口格式是什么？分页是前端还是后端？」
3. 你确认后开始写
4. 写完说：/code-review --fix 自动审查+修复

看到什么算成功：Claude 列出一个文件清单，说"我打算创建以下文件：……，修改以下文件：……，你确认吗？"，你说"确认"它就开始写。

卡住了：如果 Claude 写完了但浏览器显示空白，说"帮我排查为什么页面是空白的，先检查控制台有没有报错"。

示例A 详细流程（照这个顺序操作）

步骤1：说需求（如上），等 Claude 列出文件清单

步骤2：Claude 问确认时，回答问题。如果不确定，就说：

你来决定最合理的方式，按业界标准做

步骤3：等 Claude 写完代码，它会告诉你"代码写完了"

步骤4：让 Claude 运行看看：

帮我启动开发服务器，确认页面能正常显示

步骤5：看到服务器运行后，Claude 会告诉你访问 http://localhost:3000 查看效果

步骤6：确认没问题后，说：

做完了，帮我做个代码审查

示例B：购物车功能（状态管理）

你说：

开发购物车功能。
要求：
- 支持加入购物车、修改数量、删除商品
- 购物车数据用 Zustand 管理（本地 + 登录后同步服务端）
- 右上角 badge 显示购物车数量
- 购物车抽屉侧边弹出
做之前先给我看一下你打算怎么设计状态管理的方案

Claude 会做什么：先给你一个方案说明（不写代码），等你点头确认后再动手。

看到什么算成功：Claude 用文字描述状态管理的思路，比如"我打算用 Zustand 的 store 存购物车数组，每次修改同步到 localStorage……"

💡 加"做之前先给我看方案"是好习惯，尤其是复杂功能，确认方向再动手。

小白提示：Zustand 是一个管理数据状态的工具，就像一个"数据仓库"，购物车里有什么商品、数量多少都存在里面，页面从里面读取数据。

示例C：支付功能（高风险，需要谨慎）

你说：

接入微信支付。先帮我调研一下微信支付 JSAPI 和 Native 
两种方式，哪种适合 H5 商城，给出推荐方案和接入步骤

Claude 调研完给出推荐后：

好，就按 JSAPI 方案，帮我搭建支付服务。
注意：密钥信息只从环境变量读，不要硬编码。
做完需要做安全审查

做完后：

/security-audit

专门检查支付模块的安全漏洞（金额校验、重放攻击、签名验证等）。

看到什么算成功：安全审查结果里没有红色 P0 级别的问题，或者 P0 问题 Claude 都帮你修复了。

小白提示：支付模块是最容易出安全漏洞的地方，一定要做安全审查再上线，这不是可选的。

示例D：后台管理（复杂，用队长模式）

你说：

队长，帮我开发后台商品管理模块。
功能包括：商品列表（搜索、筛选、批量操作）、
商品新增/编辑（多规格SKU、图片上传到OSS）、
上下架操作、库存管理。
用 React 的 ant design 5 做 UI

触发"队长"模式后，Claude 会：
1. 先分析拆分成多个子任务
2. 能并行的任务同时做（列表页、表单页、API 接口可以并行）
3. 每个子任务完成后做验证
4. 全部完成后给汇总报告

看到什么算成功：Claude 先给你一个任务拆分清单，比如"我把这个模块拆成以下 5 个子任务：……然后并行做 1、2、3，做完再串行做 4、5"。

卡住了：如果 Claude 好像卡住不动了，说"继续，你做到哪一步了？"

TDD 开发模式（写测试再写代码）

适合： 核心业务逻辑、支付计算、库存扣减等不能出错的功能

你说：

用 TDD 模式开发订单价格计算逻辑。
规则：商品单价 × 数量 + 运费，
满 199 免运费，优惠券直减，最终价格不能低于 1 分钱

Claude 会严格走：
1. 红：先写失败测试（覆盖各种边界情况）
2. 绿：写最小实现让测试通过
3. 重构：整理代码，测试仍然通过

看到什么算成功：Claude 先给你一个测试文件，里面有很多 test(...) 函数，然后说"现在运行测试，全部失败，这是正常的"。接着写实现代码，最后说"测试全部通过"。

小白提示：TDD（测试驱动开发）看起来是多此一举——先写测试再写代码——但这样做出来的功能特别可靠，因为测试覆盖了所有边界情况。

🔍 第4阶段：代码质量保障

这一阶段目标：每做完一块功能就检查一次质量，上线前做全面扫描。

日常代码审查

每做完一个模块就审查：

/code-review --fix

Claude 会审查最近改动，发现问题直接修复。

看到什么算成功：Claude 输出一个审查报告，有问题的地方会标记，没有问题的地方会说"这部分没有问题"。如果有问题，Claude 会直接修复并告诉你改了什么。

卡住了：如果 Claude 修复后还有问题，说"还有问题，继续修"。

深度安全扫描

完整功能做完后：

/security-audit

或者：

帮我做一次完整的安全审计，重点检查：
SQL 注入、XSS、CSRF、支付逻辑漏洞、API 鉴权

Claude 会用 OWASP Top 10 框架逐项检查，给出分级报告（P0/P1/P2）。

看到什么算成功：Claude 给出一个分级清单，P0（必须修）没有项目，或者有的话 Claude 帮你修完了。

小白提示：
- P0 = 严重漏洞，上线前必须修
- P1 = 重要问题，建议修
- P2 = 优化建议，可选

多视角评审（重大功能上线前）

/judge-panel

或者：

帮我用多个视角审查支付模块：
① 安全视角：有没有漏洞
② 性能视角：高并发下会不会出问题
③ 边界视角：异常情况处理完整吗

看到什么算成功：Claude 用三种视角各自出一份小报告，最后给你一个综合结论。

代码简化

感觉某个文件写得太复杂：

/simplify

Claude 自动找到可以简化的地方，在不改功能的前提下精简代码。

全面 QA 验收

/ultraqa

Claude 会系统性地验证功能是否符合需求，包括：
- 功能测试
- 边界场景
- 错误处理
- 用户体验

看到什么算成功：Claude 逐条验证需求列表，每条给出"通过/不通过"，不通过的会帮你修。

📚 第5阶段：文档生成

这一阶段目标：做完代码，给自己和团队留下清晰的文档，以后接手或维护不用从头看代码。

技术架构说明文档

帮我写一份商城的技术架构说明文档，
读者是要接手项目的新工程师。
包含：架构概览、各模块说明、关键数据流、
部署架构、开发规范

看到什么算成功：Claude 输出一份有目录的 Markdown 文档，可以直接保存成 ARCHITECTURE.md。

API 接口文档

🚨 避坑指南

这些是最常见的踩坑，照着做能少走很多弯路。

坑1：直接说"帮我做个商城"

❌ 太模糊，Claude 会做出一个跟你想的完全不同的东西

✅ 正确做法：先需求分析（用 deep-interview），然后一个功能一个功能做

坑2：让 Claude 一次做太多

❌ "帮我把整个商城前端都做完"

✅ 正确做法：按功能模块拆分，每次做一个，验证完再下一个

坑3：代码改乱了不知道怎么撤回

遇到这种情况：

/rewind

或者：把你刚才的改动都撤回

也可以用 git 撤回（如果有 commit 的话）：

git checkout .

卡住了：如果改动太多撤不干净，说"帮我找到最近一次能正常运行的代码状态，把我的改动全部还原"。

坑4：Claude 把测试删掉/改掉让测试通过

如果你发现 Claude 改了测试而不是改实现：

不对，测试不能改。你要修的是实现代码，不是测试。撤回重来

Claude 有内置规则禁止这样做，但如果出现了，这样纠正。

坑5：Claude 说"应该可以了"但没验证

帮我实际运行一下，给我看运行结果

要求 Claude 给实际执行的证据，不接受"应该没问题"。

这很重要：Claude 有时候会说"应该可以了"但没有真正运行验证，你要主动要求它给你看实际运行的截图或输出，这叫"verify-before-assert"（断言前先验证）。

坑6：同一个 bug 改了好几次还没修好

同一个问题改了好几次还没修好，
帮我系统性排查一下根因，
不要先修，先把根因分析清楚再说方案

或者触发：

又报错了

→ 触发 systematic-error-analysis

看到什么算成功：Claude 给你一个根因分析，说"这个 bug 的根本原因是……不是……"，找到根因后再修，一次就能修好。

坑7：上下文塞满了无关内容

长期开发同一个项目，推荐每天/每个大模块 /clear 开新会话，
把当前关键背景（技术栈、当前在做什么）简单说一遍，比在一个会话里撑几百轮效果好。

坑8：小白常见误解

误解 实际情况
Claude 会记住所有对话 每次 /clear 后记忆清空，靠 CLAUDE.md 和 auto-memory 跨 session
代码生成了就能直接用 还需要安装依赖（npm install）、配置环境变量、做数据库迁移
一次生成整个项目 不现实，分模块做，每次做一个功能
Claude 说完成了就真完成了 要让 Claude 实际运行，给你看运行结果
报错了让 Claude 猜原因 把完整错误日志粘给 Claude，不要只说"报错了"

🆕 用最新能力做项目（实战示范）

这一节示范怎么在商城项目里用 Claude Code 的工程化能力——避坑、提速、保质量。

1. 重大技术选型前：让外部模型反驳，避免踩坑

场景：你准备在商城里用 Redis 做会话管理，但不确定这是不是最好的方案。

做法：

ai-panel "商城用户会话管理，我打算用 Redis 存 session token，
TTL 设 7 天，这个方案有没有问题？
特别是安全性和扩展性方面，帮我找出潜在的坑"

Claude 会做什么：调用 ai-panel 把这个问题同时发给多个外部模型（Kimi、MiniMax 等），让它们从不同角度反驳这个方案，找出你没想到的问题。

看到什么算成功：Claude 给你汇总多个模型的反馈，比如"两个模型都提到了 Redis 单点故障问题，建议考虑 Redis Sentinel；另外有一个模型指出 7 天 TTL 对安全性要求高的场景偏长"。

为什么这样做：Claude 本身有知识盲点，让多个不同来源的 AI 来挑战你的方案，能找到 Claude 单独分析时容易漏掉的问题。这叫"异构对抗验证"。

实际例子——商城选型场景：

ai-panel "商城图片上传，我考虑三个方案：
① 直接传到自己的 Node.js 服务器再转存 OSS
② 前端直传 OSS（带签名）
③ 用 Cloudflare R2 替代 OSS
从成本、性能、安全、运维难度四个维度反驳每个方案，
找出各自最大的坑"

2. 说"队长，火力全开"让 Claude 全权统筹一个功能

做法：

1. 拆解成子任务（数据库 → 后端 API → 前端状态 → 前端 UI → 联调）
2. 能并行的同时做（数据库迁移和后端接口可以并行）
3. 每个子任务完成后运行测试验证
4. 全部做完给你一份汇总

看到什么算成功：

  [串行] 任务3：Cart API 接口（依赖任务1）

卡住了：如果 Claude 卡在某个子任务，说"任务X卡住了，跳过继续做任务Y，X的问题稍后我来处理"。

3. 验证 API 接口：curl http://localhost:3000/api/products/1 能返回正确数据
把每步的实际输出给我看，不要只说"应该可以"

Claude 会做什么：实际运行这些命令，把终端输出给你看。

看到什么算成功：

  ✅ API 返回正确数据
